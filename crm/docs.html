Overview

CRM Lead Management for DNC CHITS

A purpose-built CRM module integrated into rosca-digital — capturing, tracking, and converting prospects into chit fund subscribers with a full interaction timeline and follow-up calendar.

🏦 Chit Fund Focus 🔗 rosca-digital Integration 📱 Multi-channel Notifications 🏢 Multi-branch 5 Implementation Phases

The existing rosca-digital system already has Subscriber, Employee, Branch, Enrollment, and task management. This CRM sits on top — leads flow in from multiple sources, get worked by marketing staff, and convert to Subscriber records in one click.

What problem does this solve?

Marketing teams currently track leads in a separate external tool (spreadsheet / CSV-based software). This creates double entry when leads become subscribers, no visibility into follow-up compliance, and no integration with operational branch staff. This CRM eliminates that gap.

3b
Phase 3b — Larger Phase

Task Management Integration

CRM follow-ups must become real task records in the existing task management system — not a bolt-on query. The task system has Processes, Stages, Stage History, Performance Reports, and Owner Performance that all need to work for CRM follow-ups automatically.

Design Decision
❌ Bolt-on approach (current)
Query leads table directly in bubbleSummary(). CRM items appear in bubble but are not real tasks — no stage history, no performance reports, no task search, no compliance tracking.
✅ Full integration (correct)
Create a CRM Follow-up Process with stages. When a follow-up date is set on a lead, create an actual tasks record. Bubble, My Tasks, Stage History, and all reports work automatically.
What needs to be built
1. Seed: CRM Follow-up Process & Stages
Add a new Process row: "CRM Follow-up". Stages: Scheduled → Due Today → Overdue → Done. Each stage has a SLA window. This is a one-time seeder (ProcessSeeder or admin panel).
2. Task Auto-creation on Follow-up Scheduled
In LeadInteraction::saved() observer — when follow_up_date is set and not null, create a tasks record: title = lead->name . ' (' . lead->lead_no . ')' process_id = CRM Follow-up process ID current_stage = 'Scheduled' assigned_to = lead->assigned_to (user_id FK) overall_due_date = follow_up_date entity_type = 'Lead', entity_id = lead->id If a task already exists for this lead's follow-up (prior scheduled FU), close it and open a new one.
3. Stage Transitions via Scheduler
New artisan command crm:advance-followup-stages runs daily at 06:00. Moves tasks:
  • Scheduled → Due Today when overall_due_date = today
  • Due Today → Overdue when overall_due_date < today and not done
This drives the bubble pulse, overdue badge, and stage history automatically.
4. Task Completion on Follow-up Done
When fu_done = true is saved on a LeadInteraction, transition the linked task to stage Done → status Completed. Logged as a stage history entry with timestamp and user_id.
5. Lead Detail — Linked Task Status
Lead detail page shows the current linked task's stage and due date inline — so the employee sees "Follow-up: Due Today · Task #7924" with a link to the task detail in rosca-digital.
6. No changes to bubble, My Tasks, or Reports
Because CRM follow-ups are real tasks records, bubbleSummary(), /tasks/my-tasks, Stage History, Process Performance, and Owner Performance all work automatically. Zero extra code for those.
⚠ Critical constraint: leads.assigned_to and tasks.assigned_to must both store user_id (FK → users). If leads.assigned_to stores an employee name string, task auto-creation will not be able to set the correct assignee and the bubble query will fail.
Deliverable: Every scheduled CRM follow-up creates a real task. Bubble, My Tasks page, Stage History, and all performance reports reflect CRM follow-up workload alongside operational tasks — no parallel system, fully unified.

Access Control

RolePermissions
MARKETING_STAFF (new) Create leads, log interactions on own leads, view own leads only. Cannot import or convert.
BRANCH_HEAD View all branch leads, reassign to any employee (any branch), convert to subscriber, import leads. Cluster heads can assign across branches.
ORG_ADMIN / SUPER_ADMIN Full access — all branches, all employees, reports, bulk import, delete leads.
SUBSCRIBER No CRM access.
Import rule: Only BRANCH_HEAD and above can import leads. During import, Branch and Employee are independent — no constraint that the employee must belong to the selected branch.

Duplicate handling

A row is a duplicate if the mobile number already exists in the system. Duplicates are skipped — shown in preview with a warning. No merge/update via CSV (update through lead detail UI only).

Column names are matched case-insensitively. Spaces, underscores, and hyphens are normalised. So mobile_no, Mobile No, and MOBILE-NO all map to the same field.
